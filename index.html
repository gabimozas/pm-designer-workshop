Framing

This is not a coding class

"The work you already do — writing specs, documenting decisions, capturing research — can become 10x more powerful when it's structured in the right place."

The problem

Engineers ask questions PMs already know the answer to. Context is scattered across Slack, Confluence, Figma, and people's heads.

The opportunity

If PMs and Designers write context where engineers actually work, everyone ships faster — and agents can help automate the rest.

Before we start

5 concepts to know

🧠

Model

The AI brain — Claude, GPT, Gemini. It reads, reasons, and writes. Everything else extends what it can do.

Example: Claude reads your initiative file and understands the constraints.

🔧

Tool

An action an agent can take — read a file, run a terminal command, post to Slack, open a PR.

Example: The agent uses a tool to read product/initiatives/cash-out.md before validating a PR.

⚡

Skill

A reusable workflow you define. Like a playbook — encode how something should always be done, then anyone can invoke it.

Example: /write-user-stories reads your initiative and generates consistent stories every time.

🔌

Plugin

An integration that connects the agent to external systems — Jira, Slack, GitHub, Figma, Confluence.

Example: A Jira plugin lets the agent create tickets directly from your initiative file, no manual input needed.

🤖

Agent

The whole system in action — a model using tools, running skills, and calling plugins to complete a goal autonomously. Built by engineers.

PMs and Designers don't build agents — they write the skills and context that make agents useful.

Before we start

How they work together

Who owns it? PM / Designer → write context: initiative files, design decisions PM / Designer → build Skills: reusable prompts that encode their thinking Engineer → configure Tools: read file, run command, check PR Engineer → wire Plugins: Jira, Slack, GitHub, Figma Engineer → build the Agent: assembles model + tools + skills + plugins All → review output and improve context over time

PMs and Designers own the context and skills — the richer they are, the smarter the agent.
Writing a good skill is a PM superpower.

Plugin vs Skill

One sentence difference

The skill tells the agent what to think.
The plugin tells the agent where to go.

SkillPlugin
What it isLogic the PM or Designer writesA door to an external system
Who creates itPM or DesignerEngineer
What it answersWhat should this output contain?Where does the agent go to act?
Without itAgent produces generic, inconsistent outputAgent can't reach Figma, Jira, or Slack
Plugin vs Skill · Example

Creating the Cash Out Confirmation screen

⚡

Skill — written by the PM

Encodes what every screen spec must contain:

/create-screen-spec

Given an initiative file + screen name:
1. User goal on this screen
2. Entry points
3. Happy path
4. Error states — list all
5. Edge cases (offline, timeout)
6. Acceptance criteria
7. Open questions for designer
🔌

Plugins — configured by engineers

Give the agent access to external systems:

Figma plugin → read component library
                check what already exists

Jira plugin   → create engineering ticket
                attach spec automatically

Slack plugin  → notify designer when ready

What happens when PM runs the skill

Runs /create-screen-spec "Cash Out Confirmation"
Agent reads the initiative file and applies the PM's skill logic
🔌

✓ Confirmation button exists in design system

The skill gave every screen a consistent standard.
The plugins made the agent reach Figma, Jira, and Slack without anyone doing it manually.

Directories · Projects · Repositories

Part 1 · Basics

A directory is just a folder

# You already use these every day 📁 My Drive 📁 Q1 Planning 📄 OKRs.doc 📄 Roadmap.xlsx 📁 Research 📄 User interviews.doc

On your computer, in Google Drive, in Notion.
A directory is just a container for files. Nothing more.

Terminal

Creating directories in the terminal

# Create a single directory $ mkdir my-initiative # Create nested directories at once $ mkdir -p product/initiatives/cash-out # See what's inside a directory $ ls -la drwxr-xr-x product/ drwxr-xr-x design/ drwxr-xr-x docs/ # Move into a directory $ cd product/initiatives

The -p flag creates all parent folders automatically — no errors if they already exist.

Part 1 · Basics

A project is a directory with purpose

📁 cash-out-redeban/ 📄 README.md ← "What this is" 📄 initiative.md ← "Why we're building this" 📄 metrics.md ← "How we measure success" 📁 research/ 📄 user-interviews.md 📁 design/ 📄 decisions.md

Think of it like a product brief folder — structured, purposeful, and understandable to anyone who opens it.

Part 1 · Basics

A repository is a shared, versioned project

Google DriveRepository

The repository is the single source of truth the whole squad syncs to — engineers, PMs, and designers alike.

How it works today vs. how it should work

Today · How it works

Loyalty Rewards — before

PM
Writes a brief in Confluence. Schedules a kickoff.
⚠ Brief has no structure. Constraints are buried in paragraphs. Success metrics are vague.
DS
Gets briefed in a meeting. Opens Figma, starts exploring.
⚠ Design decisions live in Figma comments. No reasoning is written down.
ENG
Gets a Figma link + a Confluence link in Slack.
⚠ Asks 12 questions over 3 Slack threads. Misses one constraint. Builds the wrong animation. Rework.
🔁
More meetings, more iterations, delayed launch.
Context was in people's heads. Now it's lost.
New way · How it should work

Loyalty Rewards — after

PM
Creates product/initiatives/loyalty-rewards.md
Problem, metrics, constraints, open questions — structured template. Committed to the repo.
DS
Creates design/decisions/DDD-012-rewards-card.md
Why a card over a modal. Animation spec. Constraints for engineers. Figma link. Committed.
PM
Runs /write-user-stories skill
Agent reads the initiative file → generates user stories. PM reviews and commits them.
ENG
Reads both files. Asks zero questions. Opens first PR.
Agent validates PR vs. initiative metrics and design constraints. Flags one gap. Engineer fixes it.

Structured context that engineers and agents can use

Part 2 · PMs

The initiative file

# Cash Out via Redeban/D1 ## Problem Statement Users need to withdraw money from their digital wallet to cash at physical Redeban/D1 points. ## Success Metrics - 15% of eligible users complete first Cash Out in 30 days - Transaction success rate above 98% ## Constraints - Max withdrawal: $500,000 COP per transaction - Redeban API: 3s response time limit ## Decisions Made - 2026-03-10: Bottom sheet over modal — less disruptive - 2026-03-14: No fee for first 3 months — growth priority
Terminal · PM Workflow

PMs in the terminal

# Create your initiative file $ mkdir -p product/initiatives $ touch product/initiatives/cash-out-redeban.md # Open it in your editor $ cursor product/initiatives/cash-out-redeban.md # Save your work to the shared repository $ git add product/initiatives/cash-out-redeban.md $ git commit -m "add cash-out initiative context" $ git push → Your context is now visible to the entire squad
Part 2 · PMs

The PM contribution loop

1
PM writes initiative file
Problem, metrics, constraints, open questions — structured markdown
2
Engineer reads it before writing code
No standup needed. No Slack thread. Context is there.
3
Engineer has fewer questions → ships faster
Constraints are known. Decisions are logged. Scope is clear.
4
PM updates file when decisions change
One update reaches everyone simultaneously
Part 2 · PMs

What structured context enables

⚡

Skill

A reusable task an AI performs using your context.

Example: /write-user-stories reads your initiative file and generates consistent user stories for any PM on the squad.

🤖

Subagent

An autonomous worker that monitors and acts without you present.

Example: initiative-validator checks every Pull Request against your initiative file automatically.

The PM's role evolves: write structured context once → agent handles first-pass validation → PM reviews exceptions, not everything.

Part 3

What Designers contribute

Experience logic that prevents rework

Part 3 · Designers

The designer's unique asset

"An engineer looks at a Figma file and sees what to build.
They rarely see why that decision was made.
So they make assumptions — and those assumptions cause rework."

Where it lives today

Figma comments, meeting notes, Slack threads, people's heads

Where it should live

A Design Decision Document (DDD) in the repository, next to the PM context

Part 3 · Designers

The Design Decision Document

# DDD-001: Bottom Sheet over Modal for Cash Out ## Decision Use a bottom sheet instead of a full modal. ## Reasoning - Tested with 8 users — modal felt abrupt and disorienting - Bottom sheet keeps background visible, reducing anxiety ## Constraints for Engineers - Sheet must not exceed 60% of screen height - Dismiss on tap outside must always be enabled - Animation: slide up 300ms ease-out ## What breaks if this is ignored Inconsistent experience → drop-off at confirmation step
Terminal · Designer Workflow

Designers in the terminal

# Create your design decisions folder $ mkdir -p design/decisions $ touch design/decisions/DDD-001-bottom-sheet.md # Also add UX constraints to the PM's initiative file $ cursor product/initiatives/cash-out-redeban.md # Commit both together — they belong to the same feature $ git add design/ product/ $ git commit -m "add UX constraints for cash-out flow" $ git push
Discussion · 3 min
"Think of a design decision you've made in the last month. Is that reasoning written down anywhere? What would happen if a new engineer joined tomorrow and had to implement that same feature?"

Most design reasoning lives only in people's heads. That's the gap this practice fills.

What becomes possible when context is structured

Part 4 · Agentic

The progression

1
Scattered context
Confluence, Figma, Slack, people's heads. Engineers ask questions → repeat.
2
Structured context — where we're going
Markdown files in the repo. PMs and Designers contribute alongside engineers.
3
Agentic development
Agents read context, generate drafts, validate PRs. PMs review exceptions only.
4
Continuous learning
Agent outputs improve context quality. The squad gets smarter automatically.
